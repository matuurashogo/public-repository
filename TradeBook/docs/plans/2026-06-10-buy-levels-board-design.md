# 買いレベルボード（BuyLevels）設計書

- 作成日: 2026-06-10
- ステータス: 設計確定（ユーザー承認済み）/ 実装は別途 Go 待ち
- 関連: TradeBook（表示）、jquants-data（価格）、VolDipSignals（通知基盤の流儀）、ExitLab（較正ループ）

---

## 1. 背景と目的

現在の投資フローは「AlphaScorer で良い銘柄を選ぶ → ウォッチリストに入れる → テクニカル目線で安くなったら買う」だが、最後の「安くなったら」が**数値基準のない裁量（チャートの節目・雰囲気）**になっている。

本機能はこれを言語化・システム化する:

1. **見える化**: ウォッチリスト各銘柄について「あといくら下がったら買いか」の目標価格を毎日計算して表示する（指値の目安）
2. **通知**: レベル到達・接近を毎日 LINE で受け取る
3. **言語化ループ**: どの基準が実際の買い判断と一致するかを売買記録との突合で特定していく（決め打ちしない）

## 2. ユーザー決定事項

| 論点 | 決定 |
|---|---|
| ゴール像 | 毎日の買い時通知 + 「あといくら下がったら買い」ボード |
| 「安い」の基準 | 明確な数値基準は現状なし → 6つの代理候補を併記し運用で較正 |
| 表示先 | TradeBook アプリ内 + LINE 通知の両方 |
| ウォッチリスト管理 | 手動のまま（`data/indicators_universe.json`）+ AlphaScorer 上位からの候補提案（Phase 2） |
| 通知時刻 | 夜（22時台・当日終値ベース）。翌朝の指値の仕込みに使う |
| 実装方式 | 案A: 既存ツール拡張（新リポジトリなし） |

## 3. アーキテクチャ

```
jquants-data（毎日 22:00 / 07:30 JST 更新）
   ↓ 既存 GitHub Actions（update-prices.yml）に1ステップ追加
tools/gen_buy_levels.py（新規）
   ↓ data/buy_levels.json（データ契約 TBK-0006）
   ├─→ TradeBook「買い時ボード」カード（アプリ内常設）
   └─→ LINE 通知（private 側スクリプト・22時台）
```

- 計算は公開リポジトリの既存指標パイプライン（`tools/gen_indicators.py` と同じ場所・同じ実行枠）に追加する
- LINE 通知は private 側（VolDipSignals `notify.py` と同じ LINE Messaging API 基盤）に置く。認証情報を公開側に持ち込まない

## 4. 買いレベル定義（初期セット・調整可能）

ウォッチリスト各銘柄について毎日6レベルを計算する。**全レベルを併記し、どれが正解かは決めつけない**。

| ID | 定義 | 意図 |
|---|---|---|
| L1 | 25日移動平均タッチ | 定番の押し目（エントリータグ「25日線タッチ」と対応） |
| L2 | 25日線 −5% | やや深い押し |
| L3 | 25日線 −8% | VolDipSignals reversal と同じ「深い凹み」 |
| L4 | 直近20日高値 −10% | 高値からの押し率（浅め） |
| L5 | 直近20日高値 −15% | 高値からの押し率（深め） |
| L6 | 直近60日安値 | サポートライン（下値の節目）の代理 |

付随情報:

- 各レベル: レベル価格 / 現在値からの距離% / 到達済みフラグ
- 銘柄ごと: **陽転フラグ**（終値 > 前日終値 = 下げ止まり確認）。「到達 + 陽転」が最も強い買いサインという読み方
- 価格は調整後終値ベース（OHLC 不要。現データで完結）

## 5. データ契約: `data/buy_levels.json`（TBK-0006 として実装時に正式化）

```json
{
  "updated": "2026-06-10",
  "stocks": [
    {
      "code": "6855",
      "close": 6950.0,
      "rebound": false,
      "levels": [
        { "id": "L1", "label": "25日線タッチ", "price": 7430.0, "dist": -0.065, "hit": true }
      ]
    }
  ]
}
```

- `dist` = (レベル価格 − 現在値) / 現在値（負 = すでに下回っている = 到達済み）
- `hit` = 現在値 ≤ レベル価格
- 丸め・桁数は TBK-0004 の流儀（価格 小数1桁 / 比率 小数4桁）に合わせる

## 6. 表示と通知

### TradeBook「買い時ボード」

- ウォッチリスト銘柄 × レベルの表。**到達 = 緑、あと3%以内 = 黄**で点灯
- 陽転フラグを銘柄行に表示（🟢到達+陽転 が最強表示）
- 読み取り専用（既存 `data/latest_prices.json` と同様、静的 JSON を読むだけ）

### LINE 通知（毎日22時台・1通）

```
📉 買い時ボード 6/10
🟢 到達: 6855 → L1 25日線タッチ(7,430) / L4 高値-10%(7,280) ※陽転待ち
🟡 接近: 6590 → L1 まであと2.1%（4,610）
変化なし: 7203, 6758, ...（省略表示）
```

- 「到達」「接近（3%以内）」のみ詳細表示。判定は当日終値、行動は翌朝の指値想定

## 7. 言語化ループ（較正・Phase 2）

1. 買い記録（TradeBook）と「その日に点灯していたレベル」を月次で突合
2. 「あなたの買いはどのレベルと一致しているか」を集計（例: 買った日の80%で L4 が点灯 → あなたの『節目』は高値−10%押し）
3. 一致率の高い基準を「主基準」に昇格させ、通知の強弱に反映する
4. ExitLab の較正ビュー（entrySnap 分布）と同じ思想・接続先

## 8. フェーズ計画

| Phase | 内容 | 規模 |
|---|---|---|
| 1 | TBK-0006 ADR + `gen_buy_levels.py` + ワークフロー組込 + TradeBook ボード + LINE 通知 | 数日 |
| 2 | AlphaScorer 上位×リスト未登録の「追加候補」週次提案 + 月次較正レポート（§7） | 1週間 |

## 9. 非スコープ

- 自動発注・証券会社 API 連携（通知と表示まで。最終判断は人）
- 日中足・板情報による判定（日足終値のみ）
- ウォッチリスト選定の自動化（銘柄を選ぶ裁量は残す。Phase 2 でも「提案」まで）
- RSI 逆算価格（「RSI が30になる価格」の算出は複雑な割に近似的。較正で必要性が見えたら追加）

## 10. リスクと割り切り

- **レベルが多すぎて迷う懸念**: 初期は6本併記だが、較正（§7）で主基準を絞っていく前提。最初から1本に絞ると言語化の機会を失う
- **終値判定の限界**: 日中にレベルを割って戻った場合は検知できない。寄り前の指値運用でカバーする想定
- **ウォッチリストの鮮度**: 手動管理のため陳腐化しうる。Phase 2 の候補提案で補う
