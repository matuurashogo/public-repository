# 連れ安バッジ 2段表示（candidate層追加）設計

- 日付: 2026-06-17
- 対象: 買い時ボードの連れ安度（tsureyasu）
- 新ADR: **TBK-0012**（TBK-0009 を Superseded）

## 背景・課題

連れ安バッジ（TBK-0009）はリリース以降、アプリ上で一度も表示されたことがない。
調査の結果、原因は表示・CI・コードのいずれの不具合でもなく、**急落イベントが一度も
成立していない**ためと判明した。

事実関係:

- コミット済み `data/buy_levels.json`（監視11銘柄）の `tsureyasu` キーは常に 0 件。
- 監視銘柄の直近5営業日下落率（r5）は最大でも **-6.06%**。発火ゲート（hard_drop -15%）まで
  約9%の開きがある。
- σルール（`r5 ≤ -3σ√5`）は本ユニバースでは閾値が **-17%〜-36%** に着地し、すべての銘柄で
  hard_drop(-15%) より深い。日次σが高い（2.5〜5.4%）ためで、σルールは先に発火する余地が
  なく実質デッドロジック。実効ゲートは常に hard_drop(-15%) 単独。

つまり連れ安度は「-15%級の急落」という稀なイベントでしか発火せず、TBK-0006 §7 の較正ループに
必要なデータが永遠に貯まらない構造になっている。

## 制約（乖離リスクの所在）

緩和の際に守るべき制約は「数字」ではなく「検証ドメイン」である。

- **H84（HYP-0015）** が検証したのは「**急落が起きた銘柄に限り**、連れ安🟢は個別急落🔴より
  リバウンドしやすい」という残差（resid）の優位性。
- `-15% / -3σ` という急落ゲート自体は HYP-0011（リバウンド台帳）由来で、入口の定義にすぎない。
- TBK-0009 §1 は「**検証範囲を超えない**」＝急落イベント成立時のみバッジを出す、と明示している。

したがって危険なのは閾値を下げること自体ではなく、**ゲートを緩めて H84 が未検証の軽い下げにまで
🟢/🔴 判定を付けてしまう**こと。これが「検証結果との乖離」の本体である。

## 方針: 「検証済みバッジ」と「較正用データ収集」を分離する

急落ゲートを2段に分け、検証済みの主張は一切いじらず、その下に観測層を足す。

- **confirmed（検証済み）**: 現行ゲート（`r5 ≤ -15%` または `r5 ≤ -3σ√5 かつ ≤ -5%`）。
  → 🟢連れ安 / 🔴個別急落 の色バッジ（H84 由来・**据置**）。
- **candidate（観測中）**: より浅い下げ（`r5 ≤ candidate_drop`）かつ confirmed 未満。
  → **色判定なし**の中立表示＋生 resid。連れ安/個別の語は使わない。

H84 の検証ドメインを侵さずに、発火頻度と較正データだけを増やせる。

## 詳細設計

### 1. 急落ゲート定義

- confirmed: 変更なし（`crash_state` の現行ロジックを維持）。
- candidate: `r5 ≤ candidate_drop` かつ not confirmed。残差が必要なため業種データ必須。
- 新パラメータ `candidate_drop = -0.08`（**暫定・較正対象**）。
  - -8% は L3（25日線-8%）と意味が揃う水準。フラット閾値とする。
  - σ連動（-2σ 等）は高ボラ株では結局深くなり頻度が出ないため採用しない。
  - 平穏時（現状最大 -6%）では発火しないが、"急落"の語義を保つ線として -8% を採用する
    （将来、較正で -6% への引き下げを検討可能）。

### 2. 出力スキーマ（`data/buy_levels.json`）

`tsureyasu` に `tier` を追加。candidate は `tag=null`。

```json
"tsureyasu": {
  "tier": "confirmed" | "candidate",
  "event": <tier == "confirmed">,
  "self_r5": -0.09,
  "sector": "3650",
  "sector_r5": -0.05,
  "resid": -0.04,
  "tag": "個別急落" | null
}
```

- `event` は後方互換のため残す（既存読み手は `event` を見る）。`tier=="confirmed"` のとき true。
- candidate は `tag=null`（検証済みの分類主張を行わない）。
- 候補にも満たない銘柄は **キー自体を出さない**（現行の後方互換を維持）。

### 3. 表示

- フロント confirmed: 現行の色バッジ据置（`tsureyasuBadge` は `event && tag` を要求するため、
  candidate には自然に色が付かない）。
- フロント candidate: 新規の中立チップ（例「👀観測 resid −4.0%」グレー、tooltip に self/sector r5）。
  色・連れ安/個別の語は使わない。
- LINE通知: confirmed のみ（据置）。candidate は通知しない（ノイズ抑制・YAGNI）。

### 4. ADR・テスト

- 新ADR **TBK-0012**: TBK-0009 を `Superseded` にし、candidate 層を加えた新データ契約として記述。
  public-repo に lint 連動があれば併せて更新。
- `tools/test_tsureyasu.py`: tier 判定（confirmed / candidate / 該当なし）、candidate は tag=null・
  resid あり、を追加。
- `tests/buy_levels.test.js`: candidate は中立チップで色バッジを出さない／confirmed は現行どおり。

### 5. 較正への接続

candidate の生 resid が JSON に蓄積 → TBK-0006 §7 較正ループで「-8%域でも連れ安(resid≈0)が
個別急落より戻すか」をデータ検証 → 保てば将来 candidate→confirmed 昇格、または主しきい確定。
乖離が測定可能になる。

## スコープ外（YAGNI）

- σ係数の変更（sigma_mult 3.0→2.0）は行わない。
- LINE への candidate 通知。
- 中間グレーを含む3値分類。
- 自動発注・証券会社 API 連携。
