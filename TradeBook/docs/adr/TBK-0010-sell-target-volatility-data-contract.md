# [TBK-0010] 保有銘柄の利確目標（ボラ連動）のデータ契約（volatility.json）

## 📜 ステータス

| 項目 | 値 |
|------|---|
| **Status** | `Superseded` |
| **Date** | 2026-06-16 |
| **Supersedes** | — |
| **Superseded by** | TBK-0011 |

> 利確幅の係数 `k` を 2.5 → 2.0 に改定（高ボラ銘柄の利確目標が遠すぎたため）。
> 現行の有効な契約は **TBK-0011** を参照。データ契約（JSONスキーマ）自体は不変で、既定パラメータのみ変更。

## ❓ コンテキスト（背景と課題）

保有カードは含み損益（％）を出すだけで「いつ売るか」の目安が無い。実運用は **+5%固定の利確**で
回しているが、(1) 伸びる銘柄の利を取りこぼす、(2) 銘柄ごとの「動きの大きさ」を考慮できない
（穏やかな株も荒い株も一律5%）という課題がある。

検討の結果、チャンデリア・トレールは最高値追従＝実質リアルタイム前提で終値・遅延データに不向き、
本物のATRは jquants-data が調整後終値のみ（高安なし）で算出不可のため、いずれも却下した。
採用したのは、銘柄ごとの **σ20（直近20日の日次リターン標準偏差・VolDipSignals / ExitLab と同定義）**
に連動した**静的な利確目標**を、保有カードに表示専用で出す方式。エントリー時に目標価格が確定するため
リアルタイム不要で、既存のσ定義に統一できる。設計の全体像は
`docs/plans/2026-06-16-sell-target-volatility-design.md` を参照。

## 💡 決定事項（Decision）

1. **生成方式は TBK-0006 の流儀を踏襲する**: 監視リスト（`data/indicators_universe.json`）の
   銘柄のみを対象に、`tools/gen_volatility.py` が jquants-data の日次株価から σ20 を事前計算し、
   GitHub Actions（update-prices.yml）で日次更新・コミットする。
2. **利確幅の定義（調整後終値ベース）**:
   ```
   利確幅 width = min( cap, max( floor, k × σ20 ) )
   利確目標価格 = 平均取得単価 × (1 + width)
   ```
   - `σ20` = 直近20日の日次単純リターン（終値[i]/終値[i-1]−1）の標準偏差（標本・ddof=1）。21本の終値が必要。
   - 既定 `k = 2.5` / `floor = 0.05` / `cap = 0.15`。σ20＝2%の標準的な銘柄で +5%（現行と一致）。
     穏やかな銘柄は floor の5%を死守、荒い銘柄は cap の15%まで利確目標を引き上げる。
3. **出力スキーマ `data/volatility.json`**（1ファイル・監視リスト全銘柄・コード昇順）:
   ```json
   {
     "updated": "2026-06-12",
     "source": "jquants-data prices (調整後終値ベース)",
     "window": 20,
     "k": 2.5,
     "floor": 0.05,
     "cap": 0.15,
     "sigma": { "6855": 0.0214, "5016": 0.0182 }
   }
   ```
   - `sigma` = 4桁コード → σ20（小数4桁）。履歴不足（21本未満）の銘柄はキーを出さない。
   - `window` / `k` / `floor` / `cap` = 利確幅の算出パラメータ。**表示側はこれを JSON から参照し、
     ハードコードしない**（TBK-0006 の `near_threshold` と同じ方針）。
4. **利確目標価格はブラウザで合成する**: 取得単価はユーザーの TradeBook にしか無いため、
   サーバはσ20までを配信し、`目標 = 平均取得単価 × (1 + width)` はアプリ側（`js/selltarget.js`）で計算する。
   到達判定（現在値 ≥ 目標）も含み益％も**終値ベース**。場中価格（TBK-0008）は現在値の参考表示のみで、
   利確判定の根拠には**しない**。
5. **読み手はアプリの保有カード**（`js/selltarget.js` ＋ `renderHoldings`）。本契約の JSON のみを入力とし、
   独自に再計算しない。σが無い銘柄は利確目標を `—`（空欄）にする劣化動作とし、JSON が無くても
   カード自体は従来どおり表示する。
6. **パラメータ（k / floor / cap / window）やσ定義の変更はスキーマ変更扱い**とし、本 ADR の改訂（新 ADR）で行う。

## 📈 結果・影響（Consequences）

- 保有銘柄ごとに「その銘柄の動きに見合った利確目標と、あと何％か」が毎日自動で数値化され、
  +5%固定の取りこぼし救済と、低ボラ銘柄の早すぎる利確の抑制を両立する。
- σ20は銘柄ごとの定数のため、1銘柄1数値・1ファイルで git 履歴への影響は軽微（indicators / buy_levels と同方式）。
- 場中判定に使わないため、ソース停止やσ欠損が起きても表示が消えるだけで、判定・通知・検証には影響しない。
