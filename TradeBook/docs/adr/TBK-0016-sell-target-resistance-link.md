# [TBK-0016] 利確目標を「min(σ連動目標, 最寄り抵抗線)」に改定（抵抗線連動）

| **Date** | 2026-07-20 |

## ❓ コンテキスト（背景と課題）

TBK-0010/0011 の利確目標は σ 連動（幅 = min(cap, max(floor, k×σ20))・k=2.0）だが、
超高ボラ銘柄では目標が遠すぎる問題が再発した（実例: σ20=5.6% の銘柄で幅 +11.2%。
ユーザーの実運用感覚では「10%上がるまで待てない」）。TBK-0011 で k を 2.5→2.0 に
下げた対処と同型の問題であり、係数調整では銘柄ごとの実情に追従できない。

一方、TBK-0014/0015 で**抵抗線**（スイング水準・そこで価格が止まりやすい実測値）が
使えるようになった。「抵抗線の手前で利確」は根拠のある実戦的なルールである。

## 💡 決定事項（Decision）

1. **利確目標価格 = min( σ連動目標, 取得単価より上の最寄り抵抗線 )**
   ```
   σ連動目標   = 平均取得単価 × (1 + min(cap, max(floor, k × σ20)))   ← TBK-0011 と同一
   抵抗線候補   = sr_levels.resistance のうち 取得単価 < 水準 のもの
   利確目標価格 = 候補があれば min(σ連動目標, min(候補)) / 無ければ σ連動目標
   ```
   - **取得単価以下の抵抗線は候補にしない**（そこで売るのは利確ではない。含み損時に
     損失確定を「目標」と表示しない安全弁）。
   - 実装は `js/selltarget.js` の `computeSellTarget(avgCost, currentPrice, sigma, params,
     resistances)`（第5引数は省略可＝省略時は従来どおり σ連動のみ・後方互換）。
2. **根拠（basis）の明示**: 戻り値に `basis: "sigma" | "resistance"` を持ち、表示は
   どちらが目標を決めたかを併記する（保有カード=目標価格の下に小さく／モーダル=ラベル）。
3. **σ のパラメータ（k=2.0 / floor=5% / cap=15%）と配信（volatility.json・TBK-0010 契約）は
   不変**。本 ADR は「目標価格の合成ルール」だけを変える。sr_levels 未配信・抵抗線なしの
   銘柄は従来どおり σ連動（劣化動作）。

## 📈 結果・影響（Consequences）

- 高ボラ銘柄でも、手前に抵抗線があれば現実的な利確目標になる（例: σ目標 +11.2% でも
  +2% 先に抵抗線があればそちらが目標）。
- 抵抗線が σ目標より遠い銘柄は挙動不変。
- 目標が「抵抗線ちょうど」になるため、実発注では数ティック手前の指値が実務的
  （アプリは表示専用・判定は従来どおり終値ベース）。
