# eval 先行ルール（成功条件を実装前に定義する）

> 出典思想: 「成功の定義が曖昧なまま開発すると、改善か劣化かを判断できない」。
> 実装より先に「何をもって成功とするか」と「どう測るか」を宣言する。

## 基本方針

新しい**データ契約・スコア軸・仮説・レポート**を作るときは、**コードを書く前に eval を定義する**。
これにより「改善か劣化か」を常にブレずに測れる。TradeBook では、この成功条件は多くの場合
ADR（`TradeBook/docs/adr/`）のデータ契約と `tests/` の assert として表現される。

## eval の 5 フィールド（共通スキーマ）

| フィールド | 内容 |
|---|---|
| **対象 (Target)** | 何を評価するか（1行） |
| **成功条件 (Success)** | 「良い」を観測可能な言葉で（曖昧語禁止） |
| **Grader (採点器)** | Code / Model / Human ＋ 具体的な器 |
| **合格閾値 (Threshold)** | 数値 / ルーブリック点 / スキーマ適合 |
| **反例 (Negatives)** | 「これは不合格であるべき」例（ゲーミング防止） |
| **状態 (State)** | Draft → Running → Shipped（測定値ログ付き） |

## Grader 3 類型（このリポジトリでのマッピング）

| 類型 | 手法 | このリポでの器 |
|---|---|---|
| **Code-based** | テスト・静的解析・指標計算 | `node --test tests/*.test.js`・`tools/test_*.py`・JSONスキーマ検証 |
| **Model-based** | ルーブリック採点 | （必要時のみ）LLM審査プロンプト |
| **Human** | レビュー・最終承認 | ユーザー承認 |

**原則**: Code を最優先（高速・安価・決定論）。足りない部分だけ Model / Human で補う。

## ティア制（重さを影響度で調整）

| ティア | 対象 | eval 要件 |
|---|---|---|
| **A（フル）** | データ契約・スコア軸など本番判断に影響 | 5 フィールド完全記述。Code-grader 必須化を推奨 |
| **B（軽量）** | レポート・ドキュメント | 成功条件＋Grader の 2 項だけでも可 |
| **C（不要）** | リファクタ・誤字・依存更新 | **既存テストが eval そのもの**。記述省略可 |

## 算術はコード／判断だけ LLM

計算・集計・絞り込み・並べ替えは**コード（JS/Python・決定論）**で行い、
ポリシー・文脈の解釈だけ LLM に担わせる。LLM に数式・加重・順位付けをさせるのは赤信号。
TradeBook の損益計算・手数料・KPI・買いレベルはすべて純粋関数／Python で決定論的に算出する。
