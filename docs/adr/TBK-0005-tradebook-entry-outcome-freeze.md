# [TBK-0005] 結果メトリクス（MFE/MAE・N日後リターン）の取引レコードへの凍結保存

## 📜 ステータス

| 項目 | 値 |
|------|---|
| **Status** | `Accepted` |
| **Date** | 2026-06-07 |
| **Supersedes** | — |
| **Superseded by** | — |

## ❓ コンテキスト（背景と課題）

`TBK-0003` で**入口側**の客観スナップショット（約定時点の `dev`/`abv`/`vol`/`rsi`/`hv`）を
取引レコードへ凍結保存した。これで「どんな条件で買ったか」は耐久的に残る。

「買い時の言語化」を進めるには、対になる**結果側**（買った後にどう動いたか）が要る。
`TBK-0004` で指標JSONに終値 `c` を足したので、終値系列から以下を計算できる:

- **MFE/MAE**: 約定後の最大含み益率 / 最大含み損率（タイミングの「伸び」と「耐えた含み損」）
- **N日後リターン**: +5 / +20営業日後の終値リターン（売却タイミングに依存しない素の成績）

入口側と同じく、これも**都度計算ではなく凍結保存**しないと、ローリング窓（約2年）から
古い約定の終値が落ちると遡って計算できなくなる。

## 💡 決定事項（Decision）

買い（`side === "買"`）トレードに、約定後の結果メトリクスを**焼き込む**。

1. **スキーマ拡張（後方互換・追加のみ）**: 買いトレードに任意フィールド `entryOutcome` を持たせる。

   ```json
   {
     "id": "…", "date": "2026-05-07", "code": "6590", "side": "買", "price": 4950, …,
     "entryOutcome": { "cost": 4950, "ret5": 0.153, "ret20": 0.047, "mfe": 0.153, "mae": -0.071, "asOf": "2026-06-04", "horizon": 20 }
   }
   ```

2. **計算（`computeEntryOutcome`・純粋関数）**:
   - **基準価格 `cost` = 実際の約定単価（`trade.price`）**。手取りの実感（含み益/含み損）に一致させる。
   - **固定ホライズン方式**: 起点（約定日の行）から **+20営業日**を窓とする。出口（売却日）には依存させない
     ＝取引間で比較でき、建玉マッチングも不要で、純粋に**エントリーのタイミング**を測れる。
   - `mfe`/`mae` = 窓内の最大/最小終値と `cost` の乖離率。`ret5`/`ret20` = +5/+20営業日後の終値リターン。

3. **取得タイミング（EODガード相当）**: **20営業日ぶんの終値が出揃った（`complete`）約定のみ凍結**する。
   未到達の間は凍結せず**暫定表示**し、翌営業日以降にデータが伸びて出揃った時点で確定させる
   （`TBK-0003` の当日EODガードと同じ「確定してから凍結」の思想）。

4. **永続化**: ローカルへ即時保存し、焼き込み時に `updatedAt` を更新。凍結が発生した時のみ
   正本(Drive)へ保存する（`TBK-0003` と同作法・サインイン中のみ）。`mergeMasters` は
   id 単位の last-write-wins で凍結版が勝つため消えずに伝播する。

5. **読み出し**: 凍結値があれば優先し、無ければ終値系列から暫定計算してフォールバック表示する。

6. `MASTER_VERSION` は据え置き（追加フィールドのみで後方互換）。

## 📈 結果・影響（Consequences）

- 約定後の値動き（MFE/MAE・N日後リターン）が取引レコードに永続し、ローリング窓から終値が落ちても保持される。
  入口側（`TBK-0003`）と合わせ、〔客観条件 → 主観タグ → 結果〕が一取引に揃い、買い時の言語化の土台になる。
- 既存の買いは、終値がまだ窓内にある**今のうちに**バックフィルされる（20営業日経過済みのものから順に確定）。
- **留意**: `c` は調整後終値、`cost` は生の約定単価。起点〜+20営業日に株式分割があると僅かにズレうる
  （短期20日では稀。`TBK-0004` 参照）。質的分析には十分。
- 保有期間ベース（出口依存）のMFE/MAE、監視リスト外銘柄の自動取り込みは本ADRのスコープ外（将来 ADR で検討）。

## 🔧 実行可能チェック（Enforcement）

- 計算ロジックは `tests/indicators.test.js`（`computeEntryOutcome` の MFE/MAE・ret5/ret20・`complete` 判定・不正入力 null）で検証。
- 凍結の透過は `tests/store.test.js`（`entryOutcome` が `mergeMasters` を透過し新しい方が勝つ・`setEntryOutcome`）で検証。
- スキーマ（`entryOutcome` の形・固定ホライズン方針）を変更する場合は本 ADR を `Superseded` にし、新 ADR を起こすこと。
