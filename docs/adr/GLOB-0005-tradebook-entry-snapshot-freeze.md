# [GLOB-0005] エントリー・スナップショットの取引レコードへの凍結保存

## 📜 ステータス

| 項目 | 値 |
|------|---|
| **Status** | `Accepted` |
| **Date** | 2026-06-07 |
| **Supersedes** | — |
| **Superseded by** | — |

## ❓ コンテキスト（背景と課題）

`GLOB-0004` で、エントリー時の客観指標（`dev`/`abv`/`vol`/`rsi`/`hv`）を
`data/indicators/<code>.json`（**直近約2年＝ローリング窓**・監視リスト銘柄のみ）として持ち、
表示のたびに約定日で引き当てる方式（`getSnapshot`）を定めた。

この「都度引き当て」方式には、長期の分析（**買い時の言語化**という最終目的）に対して
構造的な弱点がある:

1. **古い約定の客観値が消える** — 約2年より前の約定は、引き当てる行がローリング窓から
   落ちて客観データを永久に取得できなくなる。
2. **監視リスト外の銘柄は客観値ゼロ** — リスト外で買った取引には特徴量が一切残らない。
3. データセットが**外部データの保持期間に依存**し、時間とともに静かに痩せる。
   これは遡って復元できない。

蓄積したデータで後から「どの条件で買うと良いか」を法則化したいので、
**エントリー時点の客観特徴量を取引レコード自体へ凍結保存**し、データセットを
自己完結・耐久化させたい。

## 💡 決定事項（Decision）

買い（`side === "買"`）トレードに、エントリー時点のスナップショットを**焼き込む**。

1. **スキーマ拡張（後方互換・追加のみ）**: 買いトレードに任意フィールド `entrySnap` を持たせる。

   ```json
   {
     "id": "…", "date": "2026-06-04", "code": "6101", "side": "買", …,
     "entrySnap": { "dev": -0.0123, "abv": false, "vol": 1.42, "rsi": 48.2, "hv": 0.243, "asOf": "2026-06-04" }
   }
   ```
   - `asOf` = 引き当てに使った指標行の日付（休場フォールバック後の実日付）。
   - データが得られない場合（監視リスト外・未取得・期間外）は**付与しない**（後の機会に再試行）。

2. **取得タイミング**: 指標プリフェッチ完了後、未凍結（`!t.entrySnap`）の買いへ
   `getSnapshot(code, date)` の結果を焼き込む（`freezeEntrySnapshots`）。
   新規の買いも既存の買い（バックフィル）も**同一経路**で処理する。
   - **EODガード**: j-quants はEODデータのため、約定**当日の日中**は前日までの行しか無い。
     そのまま凍結すると前日値を誤って固定してしまう。よって**約定日ぶんが出揃ってから**
     （`latestIndicatorDate(code) >= 約定日`、判定は純粋関数 `isEntryDataReady`）凍結する。
     それまでは凍結せずライブ引き当てで暫定表示し、翌日以降に当日値で確定させる。

3. **永続化**: ローカルキャッシュへ即時保存し、焼き込み時に `updatedAt` を更新する。
   さらに**凍結が発生した時のみ正本(Drive)へ保存**する（サインイン中のみ送信、未サインインは
   ローカルのみ）。これにより、アプリを開くだけで正本へ確定し、端末キャッシュ依存にならない。
   競合は `saveMaster` の `ConflictError`→マージ再保存で吸収。`mergeMasters` は id 単位の
   last-write-wins で、`updatedAt` を更新した凍結版が勝つため消えずに伝播する。

4. **読み出し**: 凍結値（`entrySnap`）があれば優先し、無ければ従来どおりライブ
   `getSnapshot` にフォールバックする（旧データの劣化動作）。

5. `MASTER_VERSION` は据え置き（追加フィールドのみで後方互換）。

## 📈 結果・影響（Consequences）

- 約定時の客観特徴量が**取引レコードに永続**し、ローリング窓から落ちても、監視リスト外でも、
  一度凍結したものは保持される。「買い時の言語化」分析の耐久的な土台になる。
- 既存の買いは、指標がまだ窓内にある**今のうちに**バックフィルされる（救えるのは今だけ）。
- 旧データ（`entrySnap` 無し）は読み出し時にライブ引き当てへ劣化動作する。
- 結果側の指標（MFE/MAE・N日後リターン）や監視リスト外の自動取り込みは本ADRのスコープ外
  （将来 ADR で検討）。

## 🔧 実行可能チェック（Enforcement）

- `tests/store.test.js`: `entrySnap` が `normalizeMaster`/`mergeMasters` を透過し、
  `updatedAt` の新しい方（凍結版）が採用されることを検証する。
- スキーマ（`entrySnap` の形・凍結方針）を変更する場合は本 ADR を `Superseded` にし、
  新 ADR を起こすこと。
