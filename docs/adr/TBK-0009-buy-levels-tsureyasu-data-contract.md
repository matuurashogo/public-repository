## ❓ コンテキスト（背景と課題）

買いレベルボード（TBK-0006）は「あといくら下がったら買いか」を数値化するが、深い押し目
（L3=25日線−8%・L5=20日高値−15%）に来た銘柄が **買ってよい連れ安なのか、触ってはいけない
個別急落なのか** を区別できない。「安く見える」だけで個別悪材料の急落を拾うと反発しない。

別リポジトリ HypoLab の検証 **H84「連れ安残差」**（事前登録 HYP-0015、`reports/H84_report.md`）が
この区別を定量化している:

> 急落イベント直後、自銘柄の下落が「同業種なみ（連れ安）」だった群は5営業日で対TOPIX **+3.72%**
> 反発し、「業種より大きく下落（個別急落）」だった群は **−1.72%**。差 **+5.44%（p<0.001）**＝Tier A最強。
> ＝「個別急落を除外し連れ安だけを拾う」リバウンドのエントリーフィルタ。

> [!NOTE]
> 当初タスクでは「連れ安度（H83）」と参照されたが、HypoLab で検証済みの連れ安度は **H84** である
> （H83 は無関係の「増担保規制ゲーム」）。本ADRは H84 を典拠とする。

本ADRは、この検証結果を買いレベルボードに **既存スキーマへの追加フィールド1つ** で取り込む。

## 💡 決定事項（Decision）

### 1. 適用範囲は「急落イベント時のみ」（検証範囲を超えない）

H84 は **急落イベント直後・5営業日リバウンド** という条件付きでのみ検証されている。平常時の
常時指標としての効果は未検証のため、**連れ安度バッジは急落イベントが成立した銘柄にのみ付与**する。
イベントでない銘柄に `tsureyasu` は出力しない（従来表示のまま）。

### 2. 急落イベント判定（HypoLab リバウンド台帳 HYP-0011 準拠・終値のみ）

監視リスト各銘柄の調整後終値系列について、最新日を評価日として次のいずれかで「急落イベント」とする:

- `r5 = 終値[t] / 終値[t-5] − 1 ≤ −15%`（hard_drop）、または
- `r5 ≤ −3σ√5`（σ = 過去60日の日次単純リターン標準偏差。急落窓の混入を防ぐため σ は末尾5営業日を
  除いた窓で算出する＝vol_shift=5）

パラメータ: `crash_window=5` / `vol_window=60` / `vol_shift=5` / `sigma_mult=3.0` / `hard_drop=−0.15`。
σ の履歴が不足する場合は hard_drop ルールのみで判定する。σ ルールは下限ガード `min_crash_drop=−0.05`
（最低5%下落）も満たす場合のみ成立させる（HypoLab が高ボラ上位1/3に母集団を限定していたのに対し、
本ボードは監視リスト全銘柄に適用するため、低ボラ株の微小変動を急落と誤認しないための代替措置）。

### 3. 連れ安度（残差）の定義

```
連れ安度 resid = 自銘柄の5日下落率 self_r5 − 同日・同33業種・ユニバース平均の5日下落率 sector_r5
```

- 業種平均は **監視リストではなくその日の全上場（ユニバース）** で取る（H84 と同条件）。
  ユニバースの軽量フィルタ: 33業種 `9999` 除外・最新終値 ≥ 100円。
- 業種分類（33業種コード S33）は jquants-data の `full/sector33_<S33>_full.parquet`（ファイル名が S33、
  `code` 列が構成銘柄）から `code→S33` を生成する（J-Quants API 不要）。
- `resid ≈ 0`（業種なみに下落）＝連れ安、`resid` が大きく負（業種より一人だけ下落）＝個別急落。

### 4. 分類（2値・暫定しきい・較正で確定）

- **連れ安🟢 / 個別急落🔴 の2値**（中間グレーは設けない）。
- 暫定しきい `tsureyasu_resid_threshold = −0.03`: `resid ≤ −0.03`（業種より3pt以上余計に下げた）→
  **個別急落**、それ以外→ **連れ安**。
- **しきいは暫定値**。生データ（`resid` / `self_r5` / `sector_r5`）を JSON に併記し、買い記録との
  突合（TBK-0006 §7 較正ループ）で主しきいを後から確定する。決め打ちしない、という本ボードの思想に従う。

### 5. 出力スキーマ（`data/buy_levels.json` への追加・後方互換）

各 stock に **任意フィールド `tsureyasu`** を追加する。急落イベントでない、または業種データが
無く分類できない銘柄では **キー自体を出さない**（既存の読み手は無視できる＝後方互換）。

```json
{
  "code": "6855",
  "close": 6950.0,
  "rebound": false,
  "tsureyasu": {
    "event": true,
    "self_r5": -0.162,
    "sector": "3650",
    "sector_r5": -0.118,
    "resid": -0.044,
    "tag": "個別急落"
  },
  "levels": [ "...（TBK-0006 のまま）..." ]
}
```

- `self_r5` / `sector_r5` / `resid` は小数4桁（TBK-0004 の流儀）。`sector` は S33 コード。
- 既存フィールド（`levels` / `dist` / `hit` / `rebound` / `close`）は一切変更しない。

### 6. 読み手（2つ・再計算しない）

- アプリ「買い時ボード」（`js/buylevels.js`）: 銘柄行の陽転↗の隣に **🟢連れ安 / 🔴個別急落** バッジ。
- private 側 LINE 通知（`BuyLevels/notify.py`）: 到達/接近行に `※連れ安` / `※個別急落＝見送り` を付記。

「**到達 + 陽転 + 連れ安**」が最強の買いサイン、「**個別急落**」は深いレベルでも見送り、という読み筋。

### 7. 劣化動作

業種マップ（`full/`）や全上場パネルが取得できない場合、`tsureyasu` を出さずに従来どおりボードを
生成する（致命的にしない）。CI（`update-prices.yml`）では jquants-data の `full/` を sparse-checkout に
追加して有効化する。

## 📈 結果・影響（Consequences）

- 深い押し目に来た監視銘柄について、その下げが連れ安か個別急落かが毎日自動で色分けされ、
  「落ちるナイフ」を避けたエントリー判断ができる。
- しきい・パラメータの変更はスキーマ変更を伴わない場合は本ADRの改訂（誤字・暫定値更新は直接編集可）、
  フィールド構成の変更は新ADRで行う。
- 業種平均のユニバース定義は H84 の簡略版（売買代金フィルタは省略）。較正で必要なら厳密化を別ADRに。
