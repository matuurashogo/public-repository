# TradeBook エントリー根拠 ＆ エントリー・スナップショット 設計

- 日付: 2026-06-04
- 対象: 公開リポジトリ（TradeBook 本体）＋ パイプライン（private-repository / jquants-data）
- 関連:
  - `docs/plans/2026-05-30-tradebook-pnl-design.md`（損益計算の土台・平均法）
  - `docs/plans/2026-05-31-tradebook-kpi-dashboard-design.md`（KPI・保有期間/塩漬け検出）
  - `VolDipSignals`（押し目シグナルの指標ロジック）

## 背景と目的

現状の TradeBook は「損益計算ツール」である。これに **トレードの意思決定を記録・検証する**
機能を足す。狙いは2つ。

1. **主観の記録**: 各売買に「なぜ入ったか／なぜ出たか」を型タグ＋メモで残す。
2. **客観での答え合わせ**: 買った日付×銘柄から、その時点の市場データ（凹みの深さ・
   出来高急増度・トレンド位置）を後から自動で引き当て、「なんとなく買ったが、データ上は
   こうだった」を可視化する。

最終的に「**どの押し目の取り方が一番勝っているか**」を、自己申告のタグだけでなく
**実データ由来の型**でも検証できるようにする。

## ユーザー決定事項（ブレインストーミングの結論）

| 論点 | 決定 |
|---|---|
| 記録形式 | 選択式タグ ＋ 自由メモ |
| タグ管理 | 初期タグをseed、アプリ上で追加・改名・削除可 |
| 売買スタイル | 1回買って1回で売る（ほぼ1:1）→ FIFO遡及は分割時の保険 |
| 客観スナップショットの軸 | トレンド位置・凹みの深さ・出来高急増度 |
| 計算場所 | Pythonで事前計算 → 銘柄別JSONを公開 → アプリが遅延取得 |
| 出口（売り）タグ | 買い・売り**両方**に付与し、両面で分析 |

## 全体像（2層構造）

| 層 | 内容 | データの正 | フェーズ |
|---|---|---|---|
| 主観層 | 買い/売りに「型タグ＋メモ」 | Google Drive（master.json）に保存・同期 | Phase 1 |
| 客観層 | 買い日付×銘柄から指標スナップショットを自動引き当て | Python公開の指標履歴（派生・ローカルキャッシュ） | Phase 2 |

客観層のデータフロー:

```
jquants-data（adj_close・trading_value）
  → GitHub Actions（VolDipSignals の指標ロジックを流用）
  → public-repository/data/indicators/<code>.json（銘柄別・直近約2年・必要列のみ）
  → アプリが「見ている銘柄だけ」遅延取得
  → 買い日付の行を引いてスナップショット導出（凹み深さ・出来高急増・トレンド位置）
```

既存の毎営業日 07:30 JST の `data/latest_prices.json` 更新パイプラインを延長する。

## ① データモデル（master.json の拡張・version 2 → 3）

取引（trade）への追加フィールド（任意・後方互換）:

- 買い（side="買"）: `entryTag: string|null`, `entryNote: string|null`
- 売り（side="売"）: `exitTag: string|null`, `exitNote: string|null`

master 直下への追加:

- `entryTags: string[]` … 買いタグ候補。初期 seed:
  `25日線タッチ`, `深押し（節目）`, `高ボラ急落リバウンド`, `ブレイク後の押し目`,
  `出来高急増の反発`, `決算・材料`, `なんとなく（裁量）`
- `exitTags: string[]` … 売りタグ候補。初期 seed:
  `利確（目標到達）`, `利確（急騰で伸びた）`, `損切り（ルール通り）`,
  `損切り（耐えきれず）`, `時間切れ・見切り`, `地合い悪化で回避`

`store.js` の方針:

- `MASTER_VERSION` を 2 → 3 に上げる。`normalizeMaster` で `entryTags`/`exitTags` が
  無ければ seed を補う（既存値は尊重・上書き禁止）。
- `normalizeTrade` は新フィールドを欠損時のみ既定 `null` で補う。旧データは破壊しない。
- `mergeMasters`（last-write-wins / tombstone）は既存ロジックのまま動作する。
  タグ配列は「master 単位」なので、record マージ後に
  `entryTags`/`exitTags` を**和集合**で統合する（端末ごとに増やしたタグを失わない）。

**スナップショットは master に保存しない**（派生データ）。`code+date` をキーに localStorage
キャッシュへ置く。理由: 既存の「正＝Drive / 派生＝ローカル計算」分離を踏襲し、同期競合と
master 肥大を避ける。データが改訂されても再取得で更新できる。

## ② 入力 UI（index.html / app.js）

- 取引フォームで **「買」選択時は entryTag チップ＋entryNote 欄**、**「売」選択時は exitTag
  チップ＋exitNote 欄**を出し分ける。タグはチップ選択式（単一選択）＋「＋新規タグ」。
- 既存取引も編集でタグ付け可能（**遡及タグ付け**）。取引履歴の各行にタグ badge を表示。
- タグ管理 UI: タグの追加・改名・削除（削除は既存取引の参照を残すか確認）。

## ③ 客観スナップショット（Phase 2）

銘柄別 JSON（`data/indicators/<code>.json`）の 1 行 = 1 営業日、直近約2年・必要列のみ:

```json
{ "date": "2026-05-30", "devMa25": -0.062, "aboveMa75": true, "volRatio20": 2.1 }
```

- `devMa25`: 25日線乖離率（凹みの深さ）
- `aboveMa75`: 75日線の上か（トレンド位置）
- `volRatio20`: 売買代金 ÷ 20日平均（出来高急増度）

指標は VolDipSignals の `add_indicators` と同一定義（`dev_short` / `ma_long` / `tv_ma20`）を
流用し、終値ベースの制約（四本値なし）を継承する。

アプリ側:

- 買い取引の `date`/`code` で該当行を引く。該当日が未公開・上場前・データ外なら「データなし」。
- 表示例: 「あなた: `なんとなく（裁量）` ／ データ: 25日線 -6.2%・75日線上・出来高 2.1倍
  ＝ 上昇トレンドの深めの押し目」。

## ④ 型別成績（分析の主役・pnl.js / charts.js）

実現損益レコードは「売り」そのもの（`calcRealized` の `records`）である点を利用して
2系統で集計する:

- **出口タグ別**: 売りに付いた `exitTag` で `records` を直接グループ化（**FIFO 不要・最も軽い**）。
- **入口タグ別／客観軸別**: 各売りの損益を **FIFO で消費した買いロットへ遡り**、そのロットの
  `entryTag` とスナップショットへ帰属させる。既存 `holdingDaysBySell` のロット突合を流用。
  1:1 売買なら厳密一致、分割時のみ株数按分。

新カード「**エントリー型別成績**」: 軸トグルで
「入口タグ別／出口タグ別／凹みの深さ別／出来高急増の有無／トレンド位置別」を切替。
各行に **売却回数・勝率・平均利益／平均損失・期待値・合計損益**（既存 KPI 定義と整合）。

客観軸のバケット定義（叩き台・config 化）:

- 凹みの深さ（devMa25）: 浅い `> -3%` / 中 `-3%〜-7%` / 深い `≤ -7%`
- 出来高急増（volRatio20）: 急増あり `≥ 1.5` / なし `< 1.5`
- トレンド位置（aboveMa75）: 上昇トレンド / 下降トレンド

これで「自己申告の型」と「データ上の型」、さらに「入口」と「出口」の両面から
"どれが一番勝っているか" を比較できる。

## ⑤ エラー処理・整合性

- スナップショット欠損: 入口/出口タグ別分析には含めるが、客観軸分析からは除外し件数を注記。
- 平均法（表示損益）と FIFO（入口タグ按分）の差は分割売買時のみ僅差 → カードに注記。
- オフライン: キャッシュ済みスナップショットで動作。未取得は「保留」。
- 保有超過売却など既存 warnings はそのまま。

## ⑥ テスト・ハーネス

- JS（`node --test TradeBook/tests/`）: FIFO 入口タグ帰属（1:1／分割）、出口タグ直接集計、
  客観バケット閾値、`normalizeMaster` のタグ seed と和集合マージ。
- Python: 指標履歴ビルダーの単体テスト（VolDipSignals 指標流用・スキーマ検証）。
- ハーネス: 変更後に `lint_check.py` / `smoke_test.py` を実行（PostToolUse Hooks）。

## ⑦ ADR

- 指標 JSON のデータ契約（`data/indicators/<code>.json` のスキーマ・更新頻度・命名）と、
  パイプライン延長は **新規グローバル ADR**（`private-repository/docs/adr/GLOB-000X`）として記録する。
  ADR 追加時は `adr.md` ルールに従い `lint_check.py` と `_ADR_LINKS` を同時更新する。

## ⑧ フェーズ分割

- **Phase 1（先行リリース・クライアント完結）**: ①②、④の入口/出口タグ別成績まで。
  パイプライン非依存で即価値。
- **Phase 2**: ③客観スナップショット（Python 公開 ＋ 遅延取得）＋ ④の客観軸分析。

## スコープ外（次段）

- ローソク足パターン・四本値依存の指標（データが終値ベースのため不可）。
- RSI／HV／シグナル一致フラグ（今回は3軸に絞る。将来 JSON 列追加で拡張可能）。
- タグの複数選択・階層化（まず単一選択で運用）。
